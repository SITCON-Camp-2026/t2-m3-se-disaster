# 需求取捨決策

> 人類改了：v1 暫定優先服務資訊整理者，並將「信心程度」改為「原文清楚度」。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

- 如果資訊整理者沒有先把來源角色、查核狀態、人工確認理由與不能直接行動原因整理清楚，行動者會更容易被畫面誤導。
- 回報者與行動者的需求都重要，但 v1 仍使用 Phase 0 原始資訊，不應先擴張成完整回報或派工流程。

## 這個使用者最想完成什麼

- 快速看懂每筆原始資訊目前缺什麼、不能相信什麼、不能直接變成任務的原因。
- 保留查看原文、AI 草稿、人類修正與整理確認之間的差異。
- 把「下一步需要誰確認什麼」說清楚，交給後續協作者時不會像已查核事實。
- 把「能整理」和「能行動」分開，避免整理結果被下一位協作者當成任務指令。

## 這個使用者最怕什麼錯誤

- 草稿或整理彙整被誤解成 verified / confirmed。
- AI 補出原文沒有的地址、需求、數量、時間或行動建議。
- 原本的「信心程度」被誤解成資料可信度或可行動程度。

## 我採用的人類修改

- 將「信心程度」改成「原文清楚度」。
- 選項改為「原文模糊 / 原文簡略 / 原文清晰」。
- 這個欄位只描述原始文字是否足夠支撐整理判斷，不代表資料已查核、已確認或可以行動。

## 觀測者新增資料與本地持久化

- 採用方向：v1 可以規劃觀測者新增資料，但只作為「本機未整理草稿」，不是正式整理後資料。
- 儲存方式：使用瀏覽器本地持久化，例如 `localStorage`，不新增後端、資料庫、外部 API 或 v1 fixture。
- 預設狀態：本機新增資料一律從 `unverified` / `needs_review` 開始，不能直接顯示為 confirmed / verified。
- 資料邊界：本機新增資料不得寫入 `src/fixtures/shared/`，也不得覆蓋 Phase 0 原始資訊。
- 流程簡化：觀測者新增時不做「是否包含個資」的判斷節點；課堂練習先專注在未查核與不能直接行動的標示。
- 人類判斷：觀測者只負責留下原始文字與不確定處；候選類型、原文清楚度、下一步仍由資訊整理者判斷。

## 我今天暫時不處理的需求

- 不做正式回報流程；觀測者新增只作為本機草稿，不代表正式通報。
- 不做志工派工、路線、地圖或現場行動決策。
- 不新增 v1 fixture，也不把 Phase 0 原始資訊搬進 shared fixtures 假裝整理乾淨。
- 不把本地持久化視為正式保存；它只是本機草稿保留。

## 我不採用的 AI 建議

- AI 建議：
  - 讓 v1 自動判斷每筆資料的行動優先級。
  - 不採用原因：優先級會暗示真實救災決策，AI 不能替人類做這種判斷。

- AI 建議：
  - 對模糊地點或缺少資訊的資料自動補足更具體描述。
  - 不採用原因：這會把推測寫成事實，也可能放入原文沒有的真實世界資料。

- AI 建議：
  - 使用外部地圖、新聞、政府資料或社群內容查證 Phase 0 資訊。
  - 不採用原因：課程限制明確禁止補真實世界資料；Release 01 也不是實作或查證階段。

## 必須保留人工確認的判斷

- 資訊是否已外部查核，不能由整理確認取代。
- 來源角色是否為當事人、現場志工、第三方轉述或未知。
- 資訊是否過期、衝突、不完整，或是否能安全公開。
- 一筆資料是否能直接變成任務或行動依據。
- 原文清楚度只能代表原始文字是否支撐整理判斷，不能代表資料已查核或已可行動。

## 今天明確不做什麼

- 不修改 `src/` 或 Phase 0 UI。
- 不新增後端、資料庫、外部 API、真實 LLM runtime API 或地圖。
- 不新增真實災情資料、真實地址、真實電話或真實人物資料。
- 不把 release pack 教材改寫或搬成標準答案。

## 我仍不確定的地方

- 「已整理資訊彙整」是否需要改名，或是否只需要更強的狀態標示。
- 每筆資料進入整理彙整前，最低限度的人類確認紀錄要包含哪些欄位。
- 「原文清楚度」是否只對資訊整理者顯示，或也要讓行動者看見。
