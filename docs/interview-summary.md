# Main Agent 訪談彙整

> Codex 草稿，待人類刪改、補充與確認。這份文件整理三個 persona sub-agent 回饋與取捨，不代表 v1 最終決策。

## 三個 persona 的共同需求

- 明確看出資料目前是否已查核。三個 persona 都需要知道一筆資訊是 confirmed / verified，還是 `needs_review` / `unverified`。
- 保留不確定性。回報者不想被迫硬填，資訊整理者不想把草稿包裝成事實，行動者不想把線索當成任務。
- 分清楚原文、AI 推測與人類整理。三個 persona 都害怕 AI 或整理者補出原文沒有的地點、數量、需求或行動建議。
- 看懂「整理完成」不是「可以行動」。三個 persona 都擔心「確認列入已整理資訊」被下一位協作者誤解成可派工。

## 三個 persona 之間的衝突需求

- 回報者想快速留下不完整資訊；資訊整理者需要足夠欄位與判斷理由，才能避免下一位協作者誤用。
- 行動者想看到簡單明確的「能不能行動」；資訊整理者則需要保留多種待確認狀態，避免過度簡化。
- 回報者希望流程低壓、不硬填；整理者與行動者需要知道來源角色與確認依據，這會增加填寫與整理成本。
- 回報者不想判斷內部欄位；整理者需要候選類型、原文清楚度、下一步等欄位來留下判斷脈絡。

## Phase 0 原型中被推翻的假設

- 原本假設：
  - 「已整理資訊彙整」只要在畫面上註明不代表外部查核，就不會被誤解。
- 被什麼回饋推翻：
  - 資訊整理者和行動者都可能看到「已整理」就降低警覺，即使旁邊還有 `needs_review` 或 `unverified`。

- 原本假設：
  - 工作台只要保留原文與草稿，就能讓下一位協作者理解目前狀態。
- 被什麼回饋推翻：
  - 行動者還需要更清楚知道「誰要確認」與「確認前不能做什麼」，否則仍可能把摘要當成任務指令。

- 原本假設：
  - 「信心程度」可以幫助行動者理解整理品質。
- 被什麼回饋推翻：
  - 行動者指出「信心：高」和 `needs_review` 同時出現時，不知道該相信哪個狀態。
- 人類修改：
  - 將欄位改稱「原文清楚度」，選項改為「原文模糊 / 原文簡略 / 原文清晰」，讓它只描述原始文字本身是否足夠支撐整理判斷。

## 最容易造成誤導的風險

- `organized_confirmed` 被看成資料已查核，而不是整理流程完成。
- `sourceType` 被誤解成可信度或官方背書。
- AI 草稿語氣太肯定，把「可能」「轉述」「不確定」寫成像事實。
- 進入「已整理資訊彙整」後，行動者忽略仍需人工確認的條件。
- 「信心程度」被看成可行動程度，蓋過查核狀態或人工確認提醒；人類修改後改用「原文清楚度」降低誤讀。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 將「整理完成」和「外部查核」放在不同欄位或不同視覺層級，避免一個確認狀態蓋過另一個。
- 在每筆資料旁顯示「下一步要確認什麼」與「確認前不能做什麼」，而不是只顯示一般性的下一步。
- 對 AI 產生或人類改寫過的摘要保留來源標示，讓使用者看出哪些是原文、哪些是整理判斷。
- 避免讓回報者面對整理者用語；例如候選類型、原文清楚度、下一步應留在整理工作區，而不是暗示回報者要判斷。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：v1 優先服務資訊整理者，讓不確定性、人工確認與判斷理由更清楚。
  - 選項 B：v1 優先服務行動者，讓能不能行動的提示更直覺。
  - 我目前傾向：選項 A，待人類確認。
  - 理由：如果整理者沒有先把查核狀態、來源角色與不能直接行動原因整理清楚，行動者看到再簡潔的畫面仍可能誤判。

- 取捨 2：
  - 選項 A：讓回報者用更自由的方式留下資訊。
  - 選項 B：要求回報時就補足角色、時間、地點與確認狀態。
  - 我目前傾向：先不做回報端完整流程，待人類確認。
  - 理由：Release 01 是需求分析；現有資料仍來自 Phase 0 原始資訊，v1 不應新增真實回報流程或後端。

- 取捨 3：
  - 選項 A：保留原本「信心程度」欄位，協助整理者描述判斷品質。
  - 選項 B：改名為「原文清楚度」，用「原文模糊 / 原文簡略 / 原文清晰」描述原文支撐程度。
  - 我目前傾向：選項 B，作為人類修改。
  - 理由：資訊整理者仍需要留下判斷依據，但行動者已明確指出「信心：高」與 `needs_review` 會混淆；「原文清楚度」比較不會被誤解成資料已查核或可以行動。

## 不應該讓 AI 自動決定的事情

- 不讓 AI 判斷資訊已經 verified / confirmed。
- 不讓 AI 自動補真實地址、人物、電話、數量、地點或救災優先順序。
- 不讓 AI 把「是否能派人行動」變成自動決策。
- 不讓 AI 自動把 `sourceType` 解讀為可信度。

## 我採用的 AI 整理

- 採用三個 persona 都重視「不確定性不能消失」的整理。
- 採用「資訊取得方式、查核狀態、整理狀態要分開」的提醒。
- 採用「AI 草稿必須留下人工確認痕跡」的整理方向。
- 採用 sub-agent 指出的新增風險：「信心程度」可能被誤讀成可行動程度。
- 採用人類修改：將這個欄位改稱「原文清楚度」，選項使用「原文模糊 / 原文簡略 / 原文清晰」。

## 我不採用的 AI 建議

- 建議：
  - 讓 v1 直接給每筆資料一個行動優先級。
  - 不採用原因：這會讓 AI 或系統替真實救災做判斷，超出課程與安全邊界。

- 建議：
  - 用地圖、外部資料或即時 API 補足地點與最新狀態。
  - 不採用原因：Release 01 是需求分析，且本專案限制不能新增外部 API、真實地圖或真實災情資料。

## 我自己補充的判斷

- v1 若要開始設計，主要價值應該是降低誤判，而不是加快派工。
- 「已整理」這個詞需要小心，因為它對不同角色可能代表不同程度的可信度。
- 目前文件應保留「待人類確認」痕跡，不能把 Codex 產生的訪談彙整當成小組最終決策。
- 前一版主 agent 模擬訪談不應算正式訪談；本版已改用三個 sub-agent 的回饋。
- 人類已補充用語決策：「信心程度」不夠直觀，改為「原文清楚度」。

## 我仍不確定的地方

- 小隊是否同意 v1 先服務資訊整理者，而不是行動者或回報者。
- 「已整理資訊彙整」是否要改名，或只靠更明確的狀態標示就足夠。
- 每筆資料進入整理彙整前，最低限度要留下哪些人工判斷理由。
- 「原文清楚度」在畫面上要如何標示，才不會被誤會成查核狀態或可行動程度。
