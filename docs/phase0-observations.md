# Phase 0 Observations

## 我們做出了什麼？

- 首頁保留原始資訊列表，並能把單筆原始資訊送到整理工作台。
- 整理工作台現在可以建立、查看、編輯、刪除、重設整理草稿。
- 使用者在工作台確認草稿後，資料會出現在「已整理資訊」分頁；這只代表整理確認，不代表外部查核完成。
- 目前 12 筆原始資訊都有保守整理草稿與已整理彙整列管，但畫面持續顯示 `needs_review` / `unverified`，避免被誤解成 verified。
- UI 改成較密集的行政表格與公告式工作台，讓下一位協作者能掃描來源、狀態、限制和人工確認註記。

## 哪些資料讓 agent 很容易整理？

- M-010 有時間、地點、物資數量、限制與下一次盤點時間，較容易整理成候選地點狀態。
- M-009 有明確時間與集合點限制，但仍提到官方公告尚未同步，因此不能當成已查核公告。
- 有明確「不再收」「請不要送」這類限制語句的資料，比只有需求名詞的資料更容易轉成摘要。

## 哪些資料讓 agent 亂補或誤判？

- M-001 容易被補成清泥任務，但原文沒有完整地址，也沒有確認需求是否仍存在。
- M-005 的 `sourceType` 是 `official_notice`，但原文說不知道截圖日期與是否官方；不能因欄位名稱就顯示為已確認。
- M-007 提到昨天名單與今天沒空，agent 很容易只抓到「可支援水電」而忽略過期風險。

## 哪些資訊不能直接變成任務？

- M-001 地址只有模糊地標，不能直接派十幾個人前往。
- M-006 同時出現「可當集合點」與「剛剛淹水不適合停留」，需要先釐清安全狀態。
- M-008 只說 A 區先不要再派人，沒有原因，不能判斷是危險、人太多或任務已完成。
- M-012 是外地家屬來電，無法確認親友位置與需求，適合先建立查證流程，不適合直接建立救援任務。

## 哪些資訊的操作者不是當事人？

- M-011 是現場志工代長者轉述，還沒有確認長者是否同意公開完整地址。
- M-012 是外地家屬替親友來電，來電者不在現場，也無法確認親友目前位置。
- 社群貼文類資料多半不是當事人直接操作，需要把來源角色標成待確認。

## 我們剛剛偷偷假設了什麼？

- 假設「整理確認」可以先存在前端 state，而不需要後端或資料庫。
- 假設 12 筆前端整理草稿是課堂示範用的保守整理，不是標準答案或正式資料契約。
- 假設「整理已確認」只代表工作台流程完成，不能被其他系統直接解讀成救災事實已確認。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要明確定義整理草稿與正式資料契約之間的轉換條件。
- 需要人工審核流程，例如誰能確認、確認時間、確認依據與撤回規則。
- 需要記錄每次修改的操作者與版本，避免整理後看不出誰改了什麼。

## 我們需要更清楚資料格式的地方

- `verificationStatus` 和「整理確認」要分開，避免把工作台確認誤解成事實已查核。
- `sourceType` 只能表示來源管道，不能單獨表示可信度。
- 草稿需要角色欄位，標示當事人、現場志工、第三方轉述或未知。

## 我們需要人工確認的地方

- 資訊是否過期，尤其是物資數量、道路封閉、支援名單。
- 模糊地點是否能安全公開或轉成可派工位置。
- 第三方轉述是否取得當事人同意，以及是否需要保護個資。
